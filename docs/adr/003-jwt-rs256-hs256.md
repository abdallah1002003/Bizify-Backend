# ADR-003: JWT Authentication with RS256 (Production) / HS256 (Development)

**Date:** 2025-09-01  
**Status:** `Accepted`  
**Deciders:** Backend Team

---

## Context

Bizify needs stateless authentication across API requests. We must choose:
1. The token format (JWT vs opaque tokens)
2. The signing algorithm (symmetric HS256 vs asymmetric RS256)
3. Token lifecycle management (access + refresh rotation with revocation)

## Decision Drivers

- **Security:** Tokens must be unforgeable, expirable, and revocable
- **Scalability:** Stateless verification — no DB hit per request in the hot path
- **Environment Parity:** Dev environment should be simple (no key management), production should be hardened (RSA keys)
- **Single-Use Revocation:** Password reset and email verification tokens must be single-use

## Considered Options

| Option | Description |
|--------|-------------|
| **JWT HS256** → **RS256** | Symmetric dev, asymmetric prod — configurable via env |
| JWT HS256 only | Simple, but same key signs and verifies (risky for distributed systems) |
| Opaque tokens + Redis | Full revocation at cost of DB/Redis hit every request |
| Session cookies + DB | Traditional sessions, not suitable for API-first architecture |

## Decision Outcome

**Chosen option:** JWT with configurable algorithm — HS256 for local development (single `SECRET_KEY`), RS256 for production (RSA private/public key pair).

### Token Types

| Type | Lifespan | Purpose |
|------|----------|---------|
| `access` | 30 min | API authentication |
| `refresh` | 7 days | Token rotation |
| `password_reset` | 15 min | Single-use password reset |
| `email_verification` | 24 h | Single-use email verification |

### Revocation Strategy

All tokens carry a `jti` (JWT ID, UUID v4). The `token_blacklist.py` module stores revoked JTIs. On logout or token compromise, the JTI is blacklisted. In-memory blacklist is used in development; Redis blacklist in production.

### Positive Consequences

- RS256 in production: private key stays on the server, public key can be distributed to micro-services for offline verification
- `jti` enables targeted revocation without invalidating all user tokens
- Separate token types prevent misuse (e.g., refresh token cannot authenticate API calls)

### Negative Consequences / Trade-offs

- RSA key pair management adds operational overhead in production (key rotation)
- In-memory blacklist is not horizontally scalable (Redis required for multi-instance)
- Short `access` token lifespan requires refresh flow implementation on clients
